## 1. 系统概述

### 1.1 系统用途

Smart AI 任务系统是一个**通用异步AI任务处理平台**，通过飞书对话界面接收用户请求，自动化完成AI任务（3D建模、音频生成、视频生成等）的提交、跟踪和结果推送。

### 1.2 核心功能

| 功能 | 说明 |

|------|------|

| 对话式交互 | 通过飞书对话接收用户任务需求 |

| 意图识别 | 自动识别用户想要的AI任务类型 |

| 需求引导 | 逐步引导用户提交必要的素材和参数 |

| 智能推荐 | 根据需求推荐最合适的API |

| 自动化处理 | 自动提交任务、跟踪状态、下载结果 |

| 消息推送 | 任务完成后自动推送结果通知 |

### 1.3 支持的任务类型

| 任务类型 | 示例API |

|----------|---------|

| 3D建模 | Meshy.AI、火山引擎（豆包/影眸/数美）、Tripo3D |

| 音频生成 | TTS、语音合成（规划中） |

| 视频生成 | 视频生成、AI剪辑（规划中） |

## 2. 设计理念

### 2.1 基础与应用分离

系统采用**分层架构**，将通用的基础设施能力（任务队列、轮询、存储、通知、日志）与上层的业务逻辑（对话理解、任务编排）分离。

**好处**：

- 通用能力只需实现一次，各任务类型复用

- 新增任务类型只需编写适配器，无需改动基础设施

- 便于维护和测试

### 2.2 数据完整追溯

所有任务相关数据（素材、成品、操作过程）完整记录，便于审计和问题排查。

### 2.3 轻量级部署

在资源有限的服务器环境下（双核4G RAM），使用 SQLite 而非 Redis 等额外服务，降低部署复杂度。

## 4. 模块说明

### 4.1 任务队列模块

#### 功能

管理任务的存储、调度、优先级。

#### 设计方案

- **队列存储**：使用 SQLite 的 tasks 表作为任务队列

- **待处理查询**：`WHERE status IN ('pending', 'queued') ORDER BY created_at ASC`

- **优先级**：按创建时间顺序，先到先处理

#### 代码示例

```python
# 获取待处理任务
def get_pending_tasks():
    return db.query("""
        SELECT * FROM tasks
        WHERE status IN ('pending', 'queued')
        ORDER BY created_at ASC
""")
```

#### 设计理由

- 任务量小（100条/天），SQLite 完全够用

- 无需额外部署服务，降低资源消耗

---

### 4.2 状态轮询模块

#### 功能

定时查询AI任务状态，处理状态变更，触发后续流程。

#### 设计方案

- **轮询方式**：后台独立进程

- **轮询间隔**：3分钟（180秒）

- **查询方式**：批量查询（按 api_id 分组，减少API调用）

- **并发处理**：单进程串行

#### 轮询逻辑

```python
while True:
    # 1. 获取所有活跃任务
    tasks = db.query("""
        SELECT task_id, api_task_id, api_id
        FROM tasks
        WHERE status IN ('pending', 'queued', 'running')
    """)

    if tasks:
        # 2. 批量查询（按api_id分组）
        api_groups = group_by(tasks, 'api_id')
        for api_id, group in api_groups.items():
            api_task_ids = [t.api_task_id for t in group]
            results = api.batch_query(api_id, api_task_ids)

            # 3. 更新状态并处理
            for task_id, status in results.items():
                if status != tasks[task_id].status:
                    db.update_status(task_id, status)
                    if status == 'succeeded':
                        download_and_notify(task_id)

sleep(180)  # 3分钟
```

#### 设计理由

- 3D生成任务通常需要几分钟，30秒查询太频繁

- 批量查询减少API调用次数，降低API压力

### 4.3 存储管理模块

#### 功能

管理输入素材和输出成品的存储、下载、分享链接。

#### 目录结构

```
tos://{bucket}/
└─ smart-ai-tasks/
    └── {task_id}/                        ← 本地任务ID（UUID）
        ├── resources/
        │   └─ {uuid}.ext                 ← 输入文件
└── {api_task_id}.ext              ← 输出文件（API返回ID）
```

#### 文件命名规则

| 文件 | 命名来源 | 说明 |

|------|----------|------|

| 输入文件 | uuid | 本地生成，用于提交API |

| 输出文件 | api_task_id | API返回，用于保存成品 |

#### URL资源处理

当用户提交的是外部URL（图片/视频/音频URL）时，统一下载保存到TOS：

处理流程：

用户提交 URL

     ↓

下载到本地 /tmp/{uuid}.{ext}

     ↓

上传到 TOS: smart-ai-tasks/{task_id}/resources/{uuid}.{ext}

     ↓

生成 presign URL 供 API 调用

     ↓

resources 表记录实际 TOS 路径

理由：

1. 素材需要永久保留，外部链接可能失效

2. 很多API需要可访问的URL，存实际文件更可靠

3. 统一处理流程，无论来源是飞书文件还是URL

resources表字段区分：

- source_type = 'feishu_file'：飞书文件

- source_type = 'url'：外部URL（下载后存实际文件）

#### 分享方式

| 类型 | 方式 | 有效期 |

|------|------|--------|

| 输入素材 | presign URL | 24小时 |

| 输出成品 | presign URL | 24小时 |

#### 清理策略

| 类型 | 策略 |

|------|------|

| 本地临时文件 | 1小时后删除 |

| TOS输入/输出 | 永久保留（数据追溯要求） |

### 4.4 通知推送模块

#### 功能

通过飞书向用户推送任务状态通知。

#### 通知类型

| 类型 | 触发 | 内容 |

|------|------|------|

| 任务已提交 | 用户提交成功 | 任务ID、预计等待时间 |

| 任务完成 | 轮询发现成功 | 下载链接 |

| 任务失败 | 轮询发现失败 | 失败原因、建议 |

| 任务超时 | 轮询超时 | 超时提示、建议重新提交 |

#### 发送方式

- **发送身份**：飞书机器人

- **发送渠道**：原路返回

  - 私聊（DM）提交 → 私聊通知

  - 群里@机器人提交 → 群里通知

#### 重试机制

- **指数退避**：1分钟 → 2分钟 → 4分钟

- **最大重试次数**：3次（配置在 settings 表）

- **失败记录**：通知发送失败记录到 operations 表

#### 消息格式

飞书消息卡片
