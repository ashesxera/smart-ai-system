# 3D建模系统架构通用化讨论记录

## 讨论时间
2026-04-20 15:52 - 16:24

---

## 背景

用户在网盘 `tos://4-ark-claw/3d-modeling-project/docs/` 中查看了 `3d-system-design-v2.0.md`（3D建模系统设计方案v2.0），该方案目前只针对3个API接口（豆包Seed3D、影眸Hyper3D、数美Hitem3D）。

---

## 讨论要点

### 1. 通用化需求提出

用户提出问题：目前设计只针对3个API接口，能否把架构改得更通用一些，所有通过飞书提交的异步API任务都使用这个系统？

### 2. 初步方案

用户进一步说明：想把**基础和应用分离**，把底层的：
- 日志管理
- 轮询状态
- 飞书通知

做成**通用组件**，上层适配各类异步API接口。

### 3. 系统现状说明

用户指出：当前系统**并没有上线**，还在架构设计阶段。

### 4. 完整需求说明

用户补充了更详细的需求：

#### 现有/规划中的任务类型

| 来源 | 状态 | 任务类型 |
|------|------|---------|
| Meshy.AI (mesh3d-new) | 已上线 | 3D建模 |
| 火山引擎 (豆包/影眸/数美) | 设计中 | 3D建模 |
| Tripo3D | 待接入 | 3D建模 |
| 音频/视频生成 | 规划中 | AI生成 |

#### 完整业务流程

```
用户飞书对话
    ↓
意图识别（3D建模？音频？视频？）
    ↓
需求收集（对话式引导）
    ├── 主图/多视图/参考图
    ├── URL/文字描述
    ├── 参数要求
    ├── 模型精度
    └── 预期花费
    ↓
智能推荐API（匹配算法）
    ↓
用户确认
    ↓
任务提交 + 数据库记录 + 返回任务ID
    ↓
后台轮询跟踪
    ↓
下载成品到网盘
    ↓
推送成功+下载地址
```

#### 数据追溯要求

| 类型 | 内容 | 存储 |
|------|------|------|
| 素材 | 输入图片/文本/URL | TOS |
| 成品 | 生成的3D模型/音频/视频 | TOS |
| 操作日志 | 提交/状态变更/下载/通知 | SQLite |

---

## 分析与建议

### 通用化方案优劣分析

#### ✅ 优点
- 代码复用：日志、通知、轮询、存储只需实现一次
- 易于扩展：新增API类型只需写"适配器"
- 维护性好：通用逻辑集中在一处
- 职责分离：基础设施团队 vs 业务适配团队可以分工
- 测试友好：通用层可以单独mock测试

#### ❌ 缺点/挑战
- 初期开发量大：需要重构现有代码
- 抽象过度风险：可能过度设计
- 复杂度增加：新人学习成本上升

### 建议的架构分层

```
┌─────────────────────────────────────────────────────────┐
│  对话理解层（应用层）                                    │
│  • 意图识别 • 需求引导 • 智能推荐                       │
├─────────────────────────────────────────────────────────┤
│  任务编排层（应用层）                                    │
│  • 任务创建 • API适配器 • 成品处理                      │
├─────────────────────────────────────────────────────────┤
│  通用基础设施层（基础件）                               │
│  • 任务队列 • 状态轮询 • 存储管理 • 通知推送 • 日志    │
├─────────────────────────────────────────────────────────┤
│  数据库层                                                │
│  • 任务表 • 素材表 • 日志表 • 配置表                    │
└─────────────────────────────────────────────────────────┘
```

### 结论

用户确认：**"任务提交 + 数据库记录 + 返回任务ID → 后台轮询跟踪 → 下载成品到网盘 → 推送成功+下载地址"** 这部分业务流程和2.0版一致，**适合做成通用基础件**。
